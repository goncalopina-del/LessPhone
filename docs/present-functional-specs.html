Revisão Crítica — O Que Estava Errado

Honestidade radical — v2.0 · Tudo implementado e corrigido

A versão original (v1.0/v1.1) tinha problemas estruturais sérios. Foram identificados e corrigidos: 5 riscos críticos estratégicos, 7 problemas importantes, e — após implementação — 19 bugs técnicos críticos adicionais. Todos estão mapeados, corrigidos, e o código está produzido. O projecto conta com 65 ficheiros prontos para configurar credenciais e submeter às lojas.

# Problema Original Impacto Real Correcção Aplicada
C1 Sessão requer toda a família
MVP dependia de adopção multi-utilizador		 Se 1 familiar recusa instalar, a feature core não funciona. App morre antes de criar hábito. Modo Solo funcional desde o Dia 1. Família é amplificador, não requisito.
C2 Paradoxo de retenção
App cujo sucesso parece abandono		 DAU baixo = App Store penaliza rankings. Reviews dizem "deixei de usar" mesmo que o produto funcione. Score semanal, Insights diários e Diário são razões legítimas para abrir a app todos os dias.
C3 Concorrência built-in ignorada
Apple Screen Time + Google Digital Wellbeing grátis		 Objecção #1 na conversão: "Já tenho isto grátis no telemóvel." Sem resposta = sem conversão. Posicionamento explícito anti-bloqueador. Paywall com comparação directa integrada.
C4 Churn D30: 94% sem "aha moment" rápido
Valor lento, churn imediato		 Trial de 21 dias é inútil se utilizador abandona no Dia 3. Sem loop completo na Sessão 1, não há Sessão 2. "Aha moment" obrigatório em <15 min: primeira sessão → score → diário → ritual. Loop completo.
C5 Detecção de distracção = surveillance tóxica
"Pedro desbloqueou o ecrã" visível a todos		 iOS bloqueia background processes. Mostrar quem usou o telemóvel gera conflito familiar, não coesão. Risco de review negativa viral. Substituído por check-in voluntário honesto. Privacidade individual protegida. Sem monitorização passiva.
I6 Billing com ponto único de falha Admin cancela → família perde tudo. Underpriced a €3,99 para 8 pessoas. Grace period 30 dias. Exportação garantida. Preço revisto com plano família premium.
I7 Sem loop viral real Convite por email = fricção alta. Sem viral = CAC alto para sempre. Convite por link (WhatsApp-first). Referral com 30 dias extra de trial.
I8 PT demasiado pequeno para sustentabilidade Tecto teórico em PT: ~€400k-1M ARR. Insuficiente para software escalável. PT como beta 3 meses. EN+ES+BR desde o início da arquitectura.
I9 Ritual Engine v1 é um switch/case "Sugestão por hora do dia" não impressiona. Primeiro contacto decepcionante = percepção comprometida. 50 rituais curados editorialmente por especialista. Qualidade > algoritmo fraco.
I10 Offline Score sem benchmark "847 pontos" comparado com quê? Número isolado não motiva. Percentil anonimizado: "Top 23% das famílias." Privado mas contextualizado.
I11 Diário = fricção não recompensada 280 chars pós-sessão = maioria ignora. Feature parece morta. 1 emoji obrigatório. Texto completamente opcional. 3 segundos de interacção.
I12 GDPR menores subestimado "Role: young" no schema não é verificação legal. Uma queixa bloqueia a app em toda a UE. Fluxo de verificação parental. DPO consultado pré-lançamento. Schema isolado para menores.

Fluxo de Detecção e Override de Língua

Como a app determina e respeita a língua do utilizador
1
App arranca → lê locale do dispositivo
Platform.localeName (ex: "pt_PT", "en_US"). Se língua suportada → usar. Se não suportada → fallback EN-US.
2
Verifica override manual em Hive
Se utilizador escolheu língua manualmente numa sessão anterior, esse valor tem prioridade sobre o locale do dispositivo.
?
Onboarding — confirmação de língua (ecrã S-01B)
No primeiro uso, após detecção automática, mostrar: "Vamos continuar em Português?" com botão de alteração. Uma decisão explícita reduz abandono por barreira linguística.
3
Locale sincronizado com Supabase após login
users.locale actualizado → Edge Functions e notificações FCM passam a usar a língua correcta. Sincronização silenciosa, sem intervenção do utilizador.
✓
Settings → "Língua da app" — override a qualquer momento
Lista de línguas disponíveis. Mudança imediata (hot reload de strings sem restart). Persistido em Hive + Supabase. Afecta: UI, notificações, emails, conteúdo de rituais e insights.

S-06 · Auth
Ecrã S-06
Autenticação — Múltiplas Camadas
/auth
Ecrã de login/registo unificado com arquitectura de segurança em 4 camadas. Apple Sign-In obrigatório por App Store Review Guidelines §4.8 — app rejeitada sem ele. Google Sign-In obrigatório para Play Store. Magic Link como fallback universal. Biometria para re-auth. MFA opcional activável pelo utilizador.
Camada 1 — OAuth Primário
  • Apple Sign-In — obrigatório iOS/macOS
  • Google Sign-In — obrigatório Android
  • Package: flutter_appauth + sign_in_with_apple
  • Scope mínimo: email + nome (Apple pode ocultar email)
  • Apple pode gerar email relay — tratar como válido
  • Supabase OAuth2 callback handler
Camada 2 — Magic Link (fallback)
  • Supabase Auth: signInWithOtp(email)
  • Link válido 1 hora, uso único
  • Deep link: present://auth/confirm?token=xxx
  • Rate limit: 3 pedidos/hora por email
  • Sem password — elimina risco de credential stuffing
Camada 3 — Biometria Local
  • Package: local_auth (FaceID / Touch ID / Fingerprint)
  • Apenas para re-auth — nunca primeiro login
  • JWT armazenado em flutter_secure_storage
  • iOS: Keychain Services
  • Android: EncryptedSharedPreferences + Keystore
  • Fallback PIN se biometria falhar 3x
Camada 4 — MFA (opcional)
  • TOTP via Supabase MFA (RFC 6238)
  • Compatível com Google Authenticator / Authy
  • Activado voluntariamente pelo utilizador nas definições
  • Recovery codes gerados no setup (8 codes)
  • Requerido para: alterar email, cancelar subscrição
Gestão de Sessão
  • JWT access token: 1h de validade
  • Refresh token: 7 dias, rotação automática
  • Refresh token invalidado ao logout explícito
  • Sessões simultâneas: máx 5 dispositivos
  • Supabase: auth.getSession() no splash
  • Session revogável remotamente (settings)
Regras de Negócio
  • Primeiro login → setup família (S-07)
  • Login seguinte → home directamente
  • Convite pendente → aceitar antes do home
  • Auth state em Riverpod global provider
  • Logout limpa: JWT + biometria + cache local
  • Conta deletada: dados removidos em 30 dias (GDPR)
MVP Core Auth 4 Camadas
Arquitectura de Segurança — 6 Camadas
🔑
L1 — OAuth
Apple Sign-In + Google Sign-In. Zero passwords. Delegação de identidade a providers de confiança.
🔐
L2 — JWT + Refresh
Access token 1h. Refresh 7 dias com rotação. Armazenado em Keychain/Keystore. Nunca em memória não protegida.
👁️
L3 — Biometria
FaceID / Touch ID / Fingerprint para re-auth. Chave privada nunca sai do Secure Enclave (iOS) ou StrongBox (Android).
🛡️
L4 — RLS PostgreSQL
Row-Level Security no motor de BD. Isolamento familiar garantido mesmo com bug de aplicação. Zero cross-family data leak possível.
🔢
L5 — MFA / TOTP
Segundo factor opcional (RFC 6238). Obrigatório para acções críticas: alterar email, cancelar subscrição, exportar dados.
🔒
L6 — TLS + Certificate Pinning
TLS 1.3 obrigatório. Certificate pinning nos endpoints críticos (auth, pagamentos). Protecção contra MITM.
Adicionalmente: Rate limiting em todos os endpoints de auth (3 tentativas/hora). Auditoria de acessos em tabela imutável. Encriptação AES-256 em repouso (Supabase storage + backups). HTTPS enforced — sem cleartext HTTP permitido. Input validation dupla: client-side (UX) + server-side obrigatória em Edge Functions. Sanitização SQL via parameterized queries (sem concatenação de strings).
