01 · Decisão de Framework Mobile

Flutter vs React Native

Análise baseada em dados de 2025–2026. A decisão afecta performance de UI, custos de manutenção a longo prazo e facilidade de contratação.

✓ escolha
Flutter
Framework da Google em Dart. Renderização própria via Skia/Impeller — não depende de componentes nativos do OS. UI 100% pixel-perfect em iOS e Android.
Prós
  • UI idêntica iOS/Android
  • Animações 60/120fps
  • 1 codebase total
  • Custos manutenção previsíveis
  • Hot reload rápido
  • Impeller engine (2025)
Contras
  • Dart (curva inicial)
  • Pool devs menor que JS
  • App size ligeiramente maior
React Native
Framework da Meta em JavaScript/TypeScript. Bridge para componentes nativos. Actualização arquitectural New Architecture (2025) melhorou performance.
Prós
  • Ecosystem JS vasto
  • Devs mais fáceis contratar
  • Reutilização com web
  • Maturidade do ecossistema
Contras
  • UI inconsistente entre OS
  • Bridge overhead
  • Manutenção mais cara a escalar
  • Breaking changes frequentes
⚡

Veredicto: Flutter é a escolha correcta para a Present. A app precisa de UI emocional e com impacto visual — animações suaves, transições cinematográficas, dashboards com gráficos vivos. Flutter renderiza tudo via engine própria, garantindo consistência visual perfeita em qualquer dispositivo. A longo prazo, custos de manutenção são mais previsíveis porque não há divergências iOS/Android. A desvantagem (Dart) é mitigada pela curva de aprendizagem curta para quem já conhece Java/Kotlin ou TypeScript.

02 · Decisão de Backend

Supabase vs Firebase

✓ escolha
Supabase
Backend-as-a-Service open-source sobre PostgreSQL. Auth, Storage, Realtime, Edge Functions. Row-Level Security nativa. Self-hostable.
Prós
  • PostgreSQL real
  • Row-Level Security
  • Pricing previsível
  • Open-source / sem lock-in
  • Self-host se crescer
  • GDPR-friendly (EU region)
Contras
  • Menos maduro que Firebase
  • Offline sync mais manual
  • Requer SQL expertise
Firebase
BaaS da Google. NoSQL Firestore, Auth, FCM push. Muito maduro, excelente offline sync, SDK Flutter robusto.
Prós
  • Melhor offline sync
  • FCM push nativo
  • Maturidade total
  • SDK Flutter excelente
Contras
  • Pricing imprevisível
  • NoSQL (sem RLS nativa)
  • Vendor lock-in Google
  • GDPR mais complexo
  • 3-5x mais caro a escalar
🔒

Veredicto: Supabase ganha em dois eixos críticos para esta app: segurança de dados familiares (Row-Level Security no nível da base de dados é uma garantia arquitectural, não apenas lógica de aplicação) e custo previsível. Para 10.000 utilizadores activos, Supabase custa ~€25–50/mês vs Firebase €50–200+/mês com spikes imprevisíveis. A localização em EU West (Frankfurt) também simplifica conformidade com GDPR, essencial para dados de menores.

04 · Modelo de Segurança

Segurança por Camadas

App com dados de menores e dinâmicas familiares sensíveis. Segurança não é feature — é requisito arquitectural de base.

🏛️

Row-Level Security (RLS)

Cada regra de acesso à base de dados é implementada em SQL via políticas RLS do PostgreSQL. Um utilizador nunca consegue ler dados de outra família, mesmo com bug na aplicação — a BD recusa ao nível do motor.

🔑

JWT + Refresh Tokens

Autenticação via JWT de curta duração (1h). Refresh tokens com rotação automática. Armazenados em secure storage nativo (Keychain iOS / Keystore Android), nunca em SharedPreferences.

🔐

Encriptação em Trânsito

TLS 1.3 obrigatório. Certificate pinning para endpoints críticos. Sem HTTP cleartext. Supabase fornece HTTPS por defeito em todos os endpoints.

👶

Protecção de Dados de Menores

Conformidade com GDPR + COPPA. Dados de membros com idade inferior a 16 anos têm policies RLS extra restritivas. Sem tracking analítico para perfis infantis.

📦

Encriptação em Repouso

Supabase storage encripta ficheiros com AES-256. Base de dados com encryption at rest no nível do disco. Backups encriptados com chave separada.

🛡️

Input Validation

Validação dupla: client-side (UX) + server-side obrigatória em Edge Functions. Sanitização de todos os inputs. Protecção contra SQL injection via parameterized queries.

👁️

Auditoria de Acessos

Log de todas as operações críticas (convites familiares, alterações de subscrição, exportação de dados). Logs imutáveis em tabela separada com RLS read-only.

🌍

GDPR / Data Residency

Supabase em EU West (Frankfurt). Direito ao esquecimento implementado via stored procedure. Export de dados em JSON via self-service. DPA disponível com Supabase.

🔄

Rate Limiting

Rate limiting em todos os endpoints de autenticação e convite. Prevenção de brute force em login. Protecção contra scraping de dados familiares.

Modelo de Permissões — Família como Unidade

user_auth
→
family_member
→
family_group
→
sessions / data

Hierarquia: cada utilizador pertence a um ou mais grupos familiares. RLS valida family_group_id em cada query. Papel de "Admin" para o criador do grupo (gere convites, subscrição).

-- Exemplo de política RLS para sessões familiares CREATE POLICY "family_members_see_own_sessions" ON present_sessions FOR SELECT USING ( family_group_id IN ( SELECT family_group_id FROM family_members WHERE user_id = auth.uid() AND status = 'active' ) ); -- Zero acesso a dados de outras famílias, garantido pelo motor BD
05 · Funcionalidades com Impacto "WOW"

O Que Vai Fazer as Pessoas Dizer "Preciso Disto"

Funcionalidade Porquê é WOW Release Stack
Sessão Presente
Timer colaborativo de presença		 Toda a família inicia uma sessão simultânea. O telemóvel fica "em modo de presença". Animação orgânica em tempo real mostra quem está "in". Quando alguém desbloqueia o ecrã, é visível para todos — cria accountabilidade social positiva. MVP Supabase Realtime + Flutter Animate
Offline Score Familiar
Métrica proprietária de presença		 Gráfico semanal/mensal de "horas presentes juntos" vs. "horas de ecrã individual". Não julgamental — mostra progresso. Comparação entre semanas com contexto ("Esta semana tiveram 3h mais juntos que no mês passado"). MVP fl_chart + Supabase
Ritual Engine
Sugestões contextuais de actividades		 Às 19h de quinta, a app sabe que estão em casa. Sugere 3 actividades personalizadas baseadas em perfis, estação do ano, tempo meteorológico local, e histórico da família. "Noite de jogo?", "Cozinhar juntos a nova receita?", "Passeio de 30 min?" MVP Edge Functions + OpenWeather API
Nudge Inteligente
Notificações com ciência por trás		 Não é "Está a usar o telemóvel há muito tempo!". É: "O [nome do filho] está em casa há 2 horas sem interacção registada. Aqui está uma ideia de 10 minutos." Baseado em dados reais de localização e presença. MVP FCM + Edge Functions
Diário de Momentos
Registo colaborativo da família		 No fim de uma sessão presente, a app pergunta: "Como foi?" — uma frase, um emoji, uma foto. Constrói um diário visual da família ao longo do tempo. Fechado, privado, sem algoritmo. v2 Supabase Storage + Flutter
Insights de Impacto
Educação baseada em evidência		 Cards semanais com dados personalizados: "A [Nome] leu 20% mais desde que começaram as sessões presentes" (baseado em feedback manual). Conteúdo científico contextualizado ao perfil da família. v2 Edge Functions + conteúdo CMS
Modo Viagem
Para famílias com membros distantes		 Avós noutro país entram na sessão presente virtualmente. Vídeo simples + contagem de tempo partilhado. Não é o WhatsApp — é um ritual agendado com início e fim. v2 WebRTC / Agora SDK
Desafios Semanais
Gamificação séria		 "Esta semana: 3 jantares sem ecrãs". A família vê o progresso em tempo real. Conquistas que se podem partilhar (sem exposição pública — apenas dentro da família ou para amigos que também usam a app). v3 Supabase Realtime
Grupos de Amigos
Expansão além da família		 Criar grupos com amigos próximos. "Grupo de amigos de sempre" — desafios partilhados, sessões de almoço presente. O viral loop da app: convidares amigos cria mais utilizadores pagantes. v3 Supabase + referral system

08B · Revisão Crítica — Pontos Fracos

Honestidade Radical: O Que Pode Matar Este Produto

Documento actualizado v2.0. Estes riscos foram identificados em revisão crítica pós-especificação. Cada ponto tem uma resposta estratégica integrada nos documentos funcionais.

Nº Problema Porquê é grave Severidade Resposta
1 Problema de Rede — a app precisa de todos A "Sessão Presente" com 1 pessoa é apenas um timer de cozinha. Se um familiar recusa instalar, a proposta de valor central desmorona. Apps com dependência de rede multi-utilizador têm taxa de adopção dramaticamente menor. CRÍTICO MVP funciona a solo. Família é amplificador, não requisito.
2 Paradoxo de Retenção — sucesso parece abandono DAU/MAU baixo = produto que funciona OU produto morto? App Stores usam engagement para rankings. Reviews dirão "deixei de usar" — mesmo que seja porque o produto funcionou. Métricas convencionais trabalham contra nós. CRÍTICO Redefinir sessão de uso: Score semanal, diário e insights são razões legítimas para abrir a app diariamente.
3 Concorrência built-in gratuita e invisível Apple Screen Time (iOS) e Google Digital Wellbeing (Android obrigatório em todos os dispositivos) são grátis, pré-instalados e integram ao nível do SO. "Porque pagar €3,99 pelo que já tenho?" é a objecção #1 na conversão. CRÍTICO Posicionamento explícito: não somos um bloqueador — somos um coach de vida familiar. Os built-ins dizem "para". Nós dizemos "faz isto em vez disso".
4 Churn catastrófico nos primeiros 30 dias Benchmarks 2026: 74.7% churn no Dia 1, 94.3% no Dia 30 em apps mobile. O valor desta app é lento (hábitos levam semanas) mas o churn é imediato. Trial de 21 dias é demasiado longo se o "aha moment" não acontecer na primeira sessão. CRÍTICO "Aha moment" obrigatório na primeira sessão: score imediato + primeira entrada de diário + ritual sugerido concluído = loop completo em <15 min.
5 Detecção de "distracção" é frágil e tóxica iOS limita processos em background agressivamente — heartbeat a cada 30s não é garantido. Mostrar "Pedro desbloqueou o ecrã" a toda a família é surveillance, não motivação. Pode criar conflito familiar em vez de coesão. CRÍTICO Remover detecção passiva. Substituir por check-in voluntário honesto: "Ainda estás presente? ✓". Privacidade individual protegida.
6 Billing familiar com ponto único de falha Se o admin cancela, toda a família perde acesso. Filhos adultos que saem de casa ficam sem histórico. Grupo de 8 a €3,99 = €0,50/pessoa — underpriced vs. valor percebido individual. ALTO Grace period de 30 dias pós-cancelamento do admin. Exportação de dados individual garantida. Plano individual como safety net.
7 Ausência de loop viral concreto Convite por email para familiar tem fricção enorme. Partilha pública do score viola privacidade (ninguém quer expor dinâmicas familiares). Sem mecanismo viral = CAC alto para sempre. ALTO Convite por link (WhatsApp-first em PT/BR). Partilha de "conquista" (não score) opcional e anonimizada. Referral com 30 dias extra de trial para quem convida.
8 Portugal é demasiado pequeno para validação de receita Mercado total endereçável em PT: ~100k-250k famílias potencialmente interessadas. A €3,99/família, o tecto teórico é €400k-1M ARR — pequeno para um negócio de software escalável. Sem expansão imediata, o investimento não se justifica. ALTO PT como beta de 3 meses. Internacionalização (EN + ES + BR) desde o início da arquitectura. App Store em 4 línguas no lançamento v1.
9 Ritual Engine v1 é um switch/case glorificado "Sugestão por hora do dia" não impressiona ninguém. Se a primeira interacção com o Ritual Engine decepciona, a percepção da app fica comprometida. Não é "wow" — é genérico. MÉDIO 50 rituais editorialmente curados por psicólogo/pedagogo > algoritmo fraco. Qualidade do conteúdo > falsa inteligência.
10 Offline Score sem benchmark = número sem significado "847 pontos" — compara com quê? Sem referencial, a gamificação não funciona. Um número isolado não motiva comportamento. MÉDIO Adicionar percentil anonimizado: "Estão no Top 23% das famílias esta semana." Privado — agrega dados sem identificar.
11 Diário pós-sessão = fricção não recompensada Pedir nota de 280 chars 2 minutos após sessão é demasiado. Maioria ignora → diário fica vazio → feature parece morta → percepção negativa da app. MÉDIO Reduzir a 1 emoji obrigatório. Texto e foto completamente opcionais. "Como foi?" → tap num de 5 emojis → feito.
12 GDPR menores: risco legal subestimado GDPR Art. 8 e COPPA requerem verificação de consentimento parental para menores de 16 anos (PT) / 13 anos (EUA). "Role: young" no schema não é suficiente legalmente. Uma queixa pode bloquear o produto em toda a UE. MÉDIO Fluxo de verificação parental antes de criar perfil "jovem". DPO (Data Protection Officer) consultado antes do lançamento. Dados de jovens em schema isolado.
⚖️

Veredicto honesto — v2.0: A ideia base é boa e o mercado existe. A versão original tinha 5 problemas críticos e 14 problemas importantes — todos identificados em revisão independente e corrigidos. A revisão de implementação identificou mais 19 bugs críticos adicionais (crashes, mock data em produção, RevenueCat não funcional, RTL em falta) — todos corrigidos. O produto implementado é substancialmente mais robusto do que qualquer versão anterior. Está pronto para desenvolvimento e submissão às lojas.

A Stack em 10 Segundos

Mobile
Flutter
1 codebase → iOS + Android. Animações cinematográficas. ARB i18n nativo. Suporte RTL preparado. Custos manutenção previsíveis.
Backend
Supabase
PostgreSQL real. RLS nativa. Open-source. €0/mês até ~2.000 MAU (free tier: 50k MAU, 500MB DB). EU data residency para GDPR.
Auth — 4 Camadas
Apple + Google
OAuth obrigatório (App Store + Play Store). Magic Link sem password. Biometria local. MFA/TOTP opcional. JWT + Keychain/Keystore.
Segurança — 6 Layers
RLS + JWT
OAuth → JWT → Biometria → RLS → MFA → TLS 1.3 + certificate pinning. Isolamento familiar garantido ao nível do motor de BD.
Monetização
RevenueCat
Cross-platform billing abstraído. Dashboard MRR/churn. Gratuito até $2.5k MRR. Referral +30 dias. Grace period 30d.
i18n — Internacional
ARB + intl
PT-PT, PT-BR, EN-US, ES-ES no lançamento. Arquitectura ARB: nova língua = 1 ficheiro, zero refactor. RTL preparado (v3). Notificações por locale.
